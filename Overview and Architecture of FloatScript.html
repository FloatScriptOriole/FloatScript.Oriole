Overview and Architecture of FloatScript

Introduction and Architecture

FloatScript is a dynamically typed, interpreted scripting language designed for simplicity and ease of use. Its execution model relies on a virtual machine (VM) that processes a compact, efficient bytecode representation of the source code. This architecture, encompassing a lexer, parser, compiler, and virtual machine, is entirely self-hosted, meaning all these tools are written in FloatScript itself. This design provides profound insights into compiler design and the recursive elegance of self-referential systems. The entire system is a layered architecture where each component builds upon the functionality of the preceding one. We will also examine the C++ implementation of the Floatscript interpreter (floatscript.oriole.cpp), which serves as the bootstrap mechanism for the entire system.

FloatScript Bytecode Architecture and Implementation

FloatScript programs are not executed directly; they are compiled into an intermediate representation known as bytecode. This bytecode is then executed by the FloatScript Virtual Machine (VM). The instruction set is designed to be lean yet powerful, enabling efficient execution of programs. The VM is a stack-based machine, performing operations by manipulating values on a central data structure: the operand stack. This design offers several advantages: simplicity of design, bytecode compactness, ease of code generation, and natural expression evaluation.

Bytecode Format and Data Structures

FloatScript bytecode is organized into 'chunks,' containers for instructions and associated data. Each chunk is a pair: `[constants_pool, code]`. The VM uses this structure to efficiently access both executable instructions and literal values.

  • Constants Pool: An array storing all literal values, such as numbers, booleans, and strings. The compiler replaces these with an index to reduce code size and provide efficient access.
  • Code Array: Contains the sequence of bytecode instructions. The VM's instruction pointer (`ip`) traverses this array, fetching and executing instructions.
  • Stack: The central data structure where most operations manipulate values. The VM is a stack-based machine, which simplifies instruction encoding and execution.
  • Globals: Global variables are stored in a hash map. Opcodes like `OP_DEFINE_GLOBAL`, `OP_GET_GLOBAL`, and `OP_SET_GLOBAL` interact with this map.
  • Call Frames: Conceptually, a call frame represents the execution context of a function call, managing the instruction pointer, local variables, and parameters for that function.

The Compiler (floatscript_compiler.fs)

The compiler translates the Abstract Syntax Tree (AST) into bytecode. It maintains state during compilation, including `current_chunk` (for bytecode), `constants_pool` (for literals), `locals` (for variable scopes), and `scope_depth`.

The compilation process involves a recursive traversal of the AST, emitting appropriate opcodes for each node. For instance, a literal node becomes an `OP_CONSTANT` instruction while declarations involve `OP_DEFINE_GLOBAL`. For control flow (if, while), the compiler uses backpatching to fill in jump offsets later. Function declarations are compiled into separate bytecode chunks, allowing for modularity.

The Lexer (floatscript_lexer.fs)

The lexer is the first phase of compilation, transforming raw source code into a stream of meaningful tokens. These tokens represent fundamental building blocks like keywords, identifiers, operators, and literals. The `floatscript_lexer.fs` file defines a comprehensive set of token types and the logic for recognizing them.

The lexer operates by scanning the input, maintaining a position in the source code, and using helper functions to efficiently identify tokens. The core logic is in `scan_token()`, which calls specialized functions like `scan_number()` or `scan_string()` based on the character it encounters.

The Parser (floatscript_parser.fs)

The parser takes the token stream from the lexer and builds a hierarchical Abstract Syntax Tree (AST). The AST captures the grammatical structure of the code, which the compiler can then use to generate bytecode. The parser is a recursive descent parser, building the tree by calling functions for each grammatical rule.

The parser defines various AST node types to represent different language constructs, from `AST_PROGRAM` to `AST_LITERAL`. The parsing process begins with `parse_program()`, which dispatches to other functions to handle specific statements and expressions. Expressions are handled using an operator-precedence approach.

How to Code in Floatscript in the Terminal

To get started, you can run the Floatscript interpreter directly from your terminal. This will launch an interactive shell where you can execute code line by line.

Running the Interpreter

To run the interactive Floatscript shell, use the following command:

/home/ubuntu/floatscript_interpreter

This will show a prompt like: "This is floatscript.flt made by: L.Luh4n4 (type 'exit' to terminate)". You can also include libraries by typing `#include <lib_name>`.

Performance and Architectural Uniqueness

FloatScript's unique architecture is centered around a stack-based virtual machine and a lean bytecode instruction set. This design leads to surprising performance benefits.

Key Architectural Strengths

  • Purity of Stack-Based VM: The stack-centric approach simplifies the VM's internal logic, making it less prone to errors and easier to implement.
  • Bytecode Compactness: The lack of explicit register operands leads to smaller, single-byte opcodes, reducing the memory footprint of compiled programs.
  • Ease of Code Generation: The compiler can directly translate expression trees into bytecode without complex register allocation, accelerating the compilation process.
  • Natural Expression Evaluation: Language constructs naturally map to stack-based evaluation, like converting infix to postfix notation.
  • Simplicity of Instruction Set: A lean, focused instruction set means minimal overhead in decoding and dispatching instructions, leading to quick execution.
  • Direct Compilation: The compiler moves directly from the AST to bytecode without multiple intermediate representations, reducing compilation time and complexity.
  • Backpatching for Control Flow: Using jump instructions with placeholder offsets streamlines the compilation process for loops and conditionals.
  • Simple Scope Management: A simple stack of local variable maps ensures correct variable resolution without complex symbol tables.
